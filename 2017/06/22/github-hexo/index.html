在执行hexo d前可以先执行hexo s，会在本地生成静态网站。当出现Hexo is running at http://localhost:4000/. Press Ctrl+C to stop.后，在浏览器中输入localhost:4000即可查看网站。

注意，在执行hexo d时可能会出现type not found: git的错误，遇到该情况时在项目根目录下执行npm install hexo-deployer-git --save命令，再执行hexo d即可。

遇到的一些坑

图片上传问题

就在写这篇文章的同时，图片上传问题就花了一些时间。主要就是路径的问题。

Hexo中图片上传主要有两种方式。一种是引用资源文件，这种方式需要在配置文件中启用post_asset_folder，它会在新建博客时在md文件的同级目录下建立一个与md文件同名的资源文件夹。然后使用markdown 语法 ![](/example.jpg)引用，或Hexo3中的asset_img标签插件。

另一种方式是使用标签形式。

由于第一种方式无法设置图片大小，于是我采用第二种方式。随之而来的就是图片路径的设置问题。在第二种方式中，图片的路径必须以/开始，根目录是项目的root路径。假如我们将路径写成/source/_post/github-hexo/sample.img这种形式，当向服务器上执行hexo deploy时，由于只会向服务器部署public文件夹，因此会找不到路径。

这里我试着在public文件夹中建立一个images文件夹，然后路径写成/images/sample.jpg，本地测试可以。但有一个问题是，当执行hexo clean清理缓存时会将public文件夹也清理，然后执行hexo generate重新生成静态文件时此前的images文件夹也会消失，这个问题如果修改hexo代码或许可以解决，但肯定是不合适的。

于是重新了解hexo的使用后，发现在source文件夹下的内容在部署时会上传到根目录下，于是我在source文件夹下建立images文件夹，在img标签的路径中使用/images/sample.jpg。这时重新生成静态文件并部署到github pages上时，项目中就多了一个images文件夹。此时，/images/sample.jpg可以引用成功。如图所示。

感悟

第一次写技术博客，折腾了一天（主要花在学习写博客= =！）。不过挺有兴致的，不知不觉就错过了饭点。写博客的确是个费时间费精力的活呐，但是收获却蛮多。为了保证正确性，要查资料看文档，将知识在脑海中更具体化。以前学东西用过即忘，写博客可以用来加强学习挺好的。首篇完成撒花～
